## Security invariants added

1. Workflow guest paths must pass both lexical containment and canonical real-path containment. Reads and writes are capped at 16 MiB, path/glob inputs at 4 KiB, and glob traversal stops once 10,000 results would be exceeded. Absolute and parent-traversing glob patterns are rejected before enumeration; escaping symlinks and Windows junctions fail closed.
2. Workspace trust data is capped at 1 MiB, 4,096 entries, and 32 KiB per absolute path; it is schema/version validated, normalized, deduplicated, lock-serialized, and persisted through a restrictive temporary file plus atomic rename. Invalid data means “untrusted”, never process termination.
3. Browser-facing requests reject unapproved origins; Fetch Metadata also rejects `Sec-Fetch-Site: cross-site` requests that omit `Origin`. Non-browser CLI/native requests without browser metadata remain compatible. Responses are non-sniffable.
4. Debug configuration output recursively redacts provider keys, OAuth secrets, every configured header and environment value, bearer tokens, sensitive command flags, and URL credentials without mutating the live config.
5. Identifier generation either uses a cryptographically secure source or throws. It never silently falls back to predictable randomness.

## Scanner policy

The external scanner should record the false-positive dispositions above at the exact reported locations. Do not globally exclude tests from secret scanning: real credentials can still leak into fixtures. Prefer context-specific suppression for domain fields and locale strings, plus a dedicated entropy/known-secret scanner across the entire repository. A clean rerun must show the two confirmed high-risk defects and the production insecure-random findings removed; unchanged lexical false positives must be marked audited rather than “fixed” through behavior-changing code churn.
